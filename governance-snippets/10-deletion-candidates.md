## 删除候选信号

维护是双向的：每次新增规则都应同步检查删除候选清单。满足以下任一即进入删除候选：

- 近 **5** 个连续 PR 都没触发该条规则，且规则覆盖的代码路径仍在演进。
- 同类问题的统一处理已下沉到 lint / hook / harness / CI，prose 在文档层成了装饰。
- 规则引用的命令、路径、API、协议帧已废弃或重命名。
- 规则在重复其他文档或 harness（如 Claude Code / Codex CLI）默认行为，无项目特异约束。

候选清单 **6 个月** 内未被新 PR 触发或被工具豁免显式引用，下次 AGENTS.md 修补时删除。删除同样适用文档引用校验 / AGENTS 结构校验命令。

## 为什么删比加难

- 加规则的人觉得"加了不亏"——但每条都吃 agent 注意力预算
- 删规则的人必须证明"删了不会坏" + 承担"将来要是又出问题"的人情债
- 制度化"删除候选信号"让删变成例行公事，而不是个人决定

## 改写建议

- 团队还没有 PR / git 工作流：把"近 5 个 PR"改为"近 1 个月"
- 团队没有 CI / lint 兜底：第 2 条删掉（用不上）
- 想更激进：把 6 个月降到 3 个月
- 想更保守：保留候选清单 1 年，期间任何复活都重置计数
